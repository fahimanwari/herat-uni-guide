| **مسیر کشف پلتفرم:** شاگرد از گوگل جستجو می‌کند («رشته کمپیوتر ساینس هرات»). اپ موبایل در گوگل ایندکس نمی‌شود و نصب اپ برای کاربر جدید مانع بزرگ است. | ⭐ **بزرگ‌ترین تغییر: استراتژی وب-اول (Web-First).** فاز اول وب‌سایت Next.js است، نه اپ Flutter. اپ موبایل به فاز ۷ منتقل شد — Backend هیچ تغییری نمی‌کند (همان REST API، اپ فقط یک کلاینت جدید است). |

**آنچه تغییر نکرد:** کل معماری Backend (FastAPI + PostgreSQL/pgvector + RAG + لایه Provider)، اصل «محتوا ۷۰٪ ارزش است»، اصل چند-دانشگاهی از روز اول، و استراتژی AI ابری. تمام تصمیمات نسخه ۲ سر جای خود باقی است.

# ۱. دید کلی پروژه

## ۱.۱ چشم‌انداز

از روز اول، برنامه **چند-دانشگاهی (Multi-Tenant)** طراحی می‌شود: همه چیز در دیتابیس به `university_id` وصل است. در آغاز فقط پوهنتون هرات فعال است.

**قانون طلایی:** هیچ جدول و هیچ صفحه‌ای بدون `university_id` ساخته نمی‌شود.

### ⭐ دامنه پوشش (نسخه ۳.۱): هر ۱۶ پوهنځی از روز انتشار

پلتفرم از روز اول انتشار باید **همه ۱۶ پوهنځی** پوهنتون هرات را داشته باشد — نه فقط کمپیوتر ساینس. عدد تایید شده: صفحه معرفی رسمی سایت (hu.edu.af/en/introduction-herat-university) صراحتاً می‌گوید «**۱۶ پوهنځی و ۷۰ دیپارتمنت**». فهرست رسمی (نام‌های دقیق دری هنگام هماهنگی تایید شود):

طب · ستوماتولوژی (طب دندان) · انجنیری · کمپیوتر ساینس · اقتصاد · حقوق و علوم سیاسی · تعلیم و تربیه · ساینس · زراعت · علوم وترنری · ادبیات و علوم بشری · ژورنالیزم · شرعیات · هنرها · علوم اجتماعی · اداره و پالیسی عامه

**استراتژی دو سطحی:** کمپیوتر ساینس «الگوی کامل» می‌شود (فاز ۱)، اما هر ۱۶ پوهنځی از همان فاز ۱ حداقل «پروفایل پایه» دارند — هیچ شاگردی با صفحه خالی روبرو نمی‌شود. تکمیل عمقی در سه موج (جزئیات: implementation.md، فاز ۱).

**مقیاس واقعی محتوا: ۷۰ دیپارتمنت/رشته.** واحد اصلی محتوا «رشته» است نه «پوهنځی» — پروفایل پایه ۷۰ رشته ≈ ۷۰ × ۲-۳ ساعت کار محتوایی. با شبکه ۱۶ رابط و فورم استاندارد کاملاً شدنی، ولی بدون رابط‌ها نه — به همین دلیل تعیین رابط‌ها جزو «قدم‌های همین هفته» است.

### ⭐ سلسله‌مراتب واقعی دیپارتمنت‌ها (نکته کلیدی نسخه ۳.۲)

ساختار سه سطحی است: **پوهنتون ← پوهنځی ← دیپارتمنت**. و دیپارتمنت‌ها دو نوع‌اند:

- **فارغ‌ده (degree):** شاگرد در کانکور انتخابش می‌کند و از آن فارغ می‌شود. **هر دیپارتمنت فارغ‌ده دیدگاه، ماموریت و آینده شغلی مختص خود دارد.** مثال پوهنځی کمپیوتر ساینس: دیتابیس/سیستم‌های معلوماتی، نتورک/تکنالوژی معلوماتی، انجنیری نرم‌افزار — فارغ دیتابیس حوزه کاری‌اش با فارغ نتورک کاملاً متفاوت است؛ سایت باید این تمایز را شفاف و برجسته نشان دهد، نه اینکه همه را «کمپیوتر ساینس» یک‌کاسه کند.
- **غیرفارغ‌ده / خدماتی (service):** فقط خدمات آموزشی به دیگران می‌دهد و **در کانکور انتخاب نمی‌شود** — مثال: دیپارتمنت «آموزش کامپیوتر» در پوهنځی CS که مضامین کامپیوتر پوهنځی‌های دیگر را تدریس می‌کند.

**پیامد در سیستم:**
1. فیلد `department_type` در دیتابیس + فیلدهای `vision`/`mission` برای هر دیپارتمنت و هر پوهنځی
2. ماشین حساب چانس، آزمون انتخاب رشته و مقایسه‌گر **فقط** روی دیپارتمنت‌های فارغ‌ده کار می‌کنند
3. UI صفحه پوهنځی دو بخش کاملاً جدا: «رشته‌های فارغ‌ده» (برجسته، Badge «قابل انتخاب در کانکور») و «دیپارتمنت‌های خدماتی» (کوچک‌تر، با توضیح) — شاگرد هرگز اشتباهاً دیپارتمنت خدماتی را رشته کانکوری فکر نکند
4. چت‌بات این تفاوت را می‌داند و در جواب توضیح می‌دهد
5. تعداد دقیق و نوع دیپارتمنت‌های هر پوهنځی را **رابط همان پوهنځی** در فورم استاندارد ثبت و تایید می‌کند (سایت رسمی این جزئیات را ندارد — بررسی شد، صفحات پوهنځی‌ها stub هستند)

## ۱.۳ دو اصل حاکم بر پروژه

1. **محتوا ۷۰٪ ارزش پروژه است.** بهترین AI هم بدون داده غنی چیزی برای گفتن ندارد. جمع‌آوری محتوا از هفته اول، موازی با کدنویسی.
2. **وب: سئو دروازه ورود است.** شاگرد از گوگل می‌آید — صفحات رشته باید Server-Render و ایندکس‌پذیر باشند. (اصل «۸۰٪ آفلاین» برای اپ موبایل در فاز ۷ پابرجاست.)

# ۲. ویژگی‌های نهایی برنامه

## ۲.۱ ویژگی‌های پایه (ایده اصلی)

معرفی دانشگاه و مقامات، پوهنځی‌ها و رشته‌ها، استادان، لابراتوارها، ویدیوی هر پوهنځی، نقشه تعاملی کمپس، اخبار + نوتیفیکیشن، FAQ، بخش کانکور، جستجو، علاقه‌مندی‌ها، سه‌زبانه (دری/پشتو/EN)، حالت آفلاین، پنل مدیریت.

## ۲.۲ قابلیت‌های هوش مصنوعی

AI Chat عمومی، مشاور انتخاب رشته، مشاور شغلی، مقایسه رشته‌ها، راهنمای پذیرش، راهنمای کمپس (با نمایش روی نقشه)، پاسخ سه‌زبانه. همه با RAG روی داده‌های واقعی دانشگاه (بخش ۹).

## ۲.۳ ویژگی‌های ارزش‌افزوده (از نسخه ۱)

⭐ **ماشین حساب چانس قبولی + کات-آف کانکور (نسخه ۳: ویژگی محوری — دکمه اصلی صفحه اول)**، داستان فارغان موفق، تقویم اکادمیک با یادآوری فعال (الگوی Pounce)، بورسیه‌ها، مقایسه‌گر رشته‌ها، Dark Mode، آنالیتیکس ادمین، اشتراک‌گذاری، فرم «سوال از دانشگاه».

## ۲.۴ ⭐ ویژگی جدید: Career Roadmap (نقشه راه شغلی)

داخل صفحه هر رشته، برای هر مسیر شغلی یک نقشه راه گام‌به‌گام تعاملی:

```
Computer Science → Backend Developer:
  Python → Git → Linux → SQL → FastAPI → Docker → پروژه واقعی → CV
```

- **پیاده‌سازی:** محتوای ثابت JSON در دیتابیس (`career_roadmaps` — بخش ۵)، نمایش به صورت stepper عمودی زیبا در Flutter. **کاملاً آفلاین، بدون AI.**
- برای هر گام: عنوان، توضیح کوتاه، منابع یادگیری رایگان (لینک یوتیوب/کورس).
- محتوای اولیه را خود شما (استاد CS) برای رشته کمپیوتر ساینس بنویسید — الگوی بقیه می‌شود.

## ۲.۵ صفحه رشته باید «غنی» باشد (Data-Driven)

وقتی کاربر روی Computer Science کلیک می‌کند، باید ببیند: عکس و ویدیو، توضیح، مضامین اصلی، آینده شغلی، بازار کار، مهارت‌های لازم، «مناسب چه کسی است»، استادان، لابراتوارها، **پروژه‌های نمونه دانشجویان**، **حوزه‌های تحقیقاتی**, **مصاحبه/داستان فارغان**، کات-آف کانکور، Career Roadmaps، و FAQ مخصوص رشته. (جداول مربوطه در بخش ۵ اضافه شد.)

## ۲.۶ ⭐ طراحی و تجربه کاربری (نسخه ۳ — اولویت درجه یک)

ظاهر حرفه‌ای و تجربه دلپذیر، شرط اعتماد کاربر است — به‌خصوص وقتی رقیب نداریم و «اولین برداشت» همه چیز است. شش اصل:

1. **هویت بصری «کاشی هراتی»:** پالت الهام‌گرفته از مسجد جامع هرات — لاجوردی (رنگ اصلی)، فیروزه‌ای (تاکید)، طلایی (CTA ویژه). هویت یکتا، نه قالب آماده.
2. **موبایل-اول:** کاربر واقعی = شاگرد با گوشی اندروید ارزان. طراحی از عرض ۳۶۰px شروع می‌شود، دسکتاپ گسترش آن است.
3. **سرعت خودِ UX است:** اینترنت افغانستان ضعیف است — بودجه سرعت 3G، تصاویر WebP، فونت سبک. سایت کند = سایت زشت.
4. **Dark Mode** از روز اول (توکن-محور، بدون هزینه اضافه).
5. **دسترس‌پذیری:** کنتراست استاندارد، رنگ هرگز تنها حامل معنا نباشد (چانس سبز/زرد/سرخ همیشه با متن و آیکن).
6. **سیستم دیزاین توکن-محور:** هیچ رنگ و اندازه‌ای مستقیم در کد — همه از توکن‌ها. تغییر سلیقه فردا = تغییر چند توکن، نه بازنویسی کامپوننت‌ها.
7. **⭐ سلسله‌مراتب شفاف:** breadcrumb «پوهنتون ← پوهنځی ← دیپارتمنت» در همه صفحات؛ رشته‌های فارغ‌ده و دیپارتمنت‌های خدماتی هرگز قاطی نشوند؛ تمایز شغلی رشته‌های همسایه (فارغ دیتابیس ≠ فارغ نتورک) در صفحه هر رشته برجسته باشد.

> جزئیات فنی (کد توکن‌ها، فهرست کامپوننت‌ها، قوانین UX) در **implementation.md — مرحله ۲.۲**.

# ۹. ⭐ هوش مصنوعی: انتخاب API، اتصال و RAG (بازنویسی کامل)

## ۹.۱ چرا مدل محلی (Ollama) برای ما جواب نمی‌دهد

1. **سرور نداریم.** VPS ارزان ما ۴GB RAM دارد؛ حتی مدل 3-4B روی آن یا اجرا نمی‌شود یا برای هر جواب ۳۰-۶۰ ثانیه طول می‌کشد. سرور GPU حداقل ۱۰۰-۳۰۰ دلار/ماه است — گران‌تر از API!
2. **کیفیت دری/پشتو.** مدل‌های کوچک متن‌باز در فارسی/دری ضعیف و در پشتو تقریباً غیرقابل استفاده‌اند. مدل‌های ابری بزرگ به‌مراتب بهترند.
3. **ادعای «بدون اینترنت» بی‌معناست** — سرور به هر حال باید آنلاین باشد تا اپ به آن وصل شود.

**اما ایده لایه Provider را نگه می‌داریم:** اگر روزی دانشگاه سرور GPU داد، `ollama.py` را کنار بقیه providers می‌گذارید و فقط `.env` عوض می‌شود.

## ۹.۲ کدام API؟ (مقایسه عملی برای شرایط شما)

| گزینه | هزینه | کیفیت دری/پشتو | پرداخت از افغانستان | توصیه |
|---|---|---|---|---|
| **Google Gemini Flash** (aistudio.google.com) | **سطح رایگان روزانه دارد**؛ نسخه پولی هم بسیار ارزان | خوب | فقط حساب Google برای شروع؛ برای سطح پولی کارت لازم است | ⭐ **بهترین نقطه شروع**: توسعه و تست کاملاً رایگان |

**استراتژی پیشنهادی (عملی و بدون ریسک):**
1. **همین امروز:** حساب رایگان Google AI Studio بسازید → کلید Gemini بگیرید → کل توسعه و تست فاز ۲ با سطح رایگان انجام می‌شود (هزینه: صفر).

3. چون OpenRouter با **فرمت API سازگار با OpenAI** کار می‌کند، یک provider آن را پوشش می‌دهد و «تغییر مدل» فقط تغییر نام مدل در `.env` است.

## ۹.۳ لایه Provider (کد واقعی برای تیم)

```python
# services/ai/provider.py
from abc import ABC, abstractmethod

class AIProvider(ABC):
    @abstractmethod
    async def chat(self, system: str, messages: list[dict],
                   max_tokens: int = 800) -> str: ...

# services/ai/providers/openrouter.py  (Gemini/Claude/GPT همه از همین می‌گذرند)
import httpx
from app.config import settings

class OpenRouterProvider(AIProvider):
    async def chat(self, system, messages, max_tokens=800):
        async with httpx.AsyncClient(timeout=60) as client:
            r = await client.post(
                "https://openrouter.ai/api/v1/chat/completions",
                headers={"Authorization": f"Bearer {settings.AI_API_KEY}"},
                json={
                    "model": settings.AI_MODEL,   # ⭐ فقط .env عوض می‌شود
                    "max_tokens": max_tokens,
                    "messages": [{"role": "system", "content": system}, *messages],
                },
            )
            r.raise_for_status()
            return r.json()["choices"][0]["message"]["content"]

# services/ai/providers/gemini.py — مشابه، با endpoint گوگل
# .env:
#   AI_PROVIDER=gemini          → برای توسعه (رایگان)
#   AI_PROVIDER=openrouter      → برای production
#   AI_MODEL=google/gemini-2.5-flash
#   AI_API_KEY=...              ← فقط روی سرور! هرگز داخل اپ Flutter!
```

## ۹.۴ RAG با pgvector (به جای Qdrant)

**چرا RAG؟** مدل درباره پوهنتون هرات چیزی نمی‌داند؛ باید قبل از هر سوال، اطلاعات مرتبط از دیتابیس خودمان به prompt چسبانده شود — وگرنه جواب اشتباه می‌سازد.

**چرا pgvector و نه Qdrant؟** همان جستجوی برداری، داخل PostgreSQL موجود، بدون سرویس جدید، بدون نگهداری اضافه. در مقیاس ما (زیر ۱۰۰هزار قطعه) کاملاً کافی است.

**خط لوله (Pipeline) — تیم دقیقاً همین را بسازد:**

```
مرحله A: ایندکس‌سازی (هر بار ادمین محتوا را تغییر داد — خودکار یا با /admin/rag/reindex)
  1. متن هر موجودیت (رشته، FAQ، راهنما، ساختمان، بعداً PDF مقررات)
     به قطعات ۲۰۰-۵۰۰ کلمه‌ای تقسیم شود
  2. هر قطعه → embedding با multilingual-e5-small (روی CPU سرور، رایگان)
  3. ذخیره در جدول rag_chunks

مرحله B: پاسخ‌دهی (هر پیام کاربر)
  1. کش Redis چک شود (هش سوال نرمال‌شده) → اگر بود، جواب فوری و رایگان
  2. embedding سوال محاسبه شود
  3. SELECT ... ORDER BY embedding <=> :q_emb LIMIT 8   ← جستجوی کسینوسی pgvector
  4. ۸ قطعه مرتبط + system prompt → ارسال به LLM API
  5. جواب → کش Redis (۷ روز) → ثبت در ai_chat_logs → برگشت به اپ
```

```python
# services/ai/embeddings.py
from sentence_transformers import SentenceTransformer
_model = SentenceTransformer("intfloat/multilingual-e5-small")  # یک بار لود می‌شود

def embed(text: str) -> list[float]:
    return _model.encode(f"query: {text}", normalize_embeddings=True).tolist()
```

**System Prompt (در prompts.py):**
```
تو دستیار هوشمند {university_name} هستی. فقط بر اساس «اطلاعات دانشگاه»
زیر جواب بده. اگر جواب در اطلاعات نبود صادقانه بگو نمی‌دانی و شماره
تماس دانشگاه را پیشنهاد کن. به زبان {language} و با لحن دوستانه جواب بده.
=== اطلاعات دانشگاه ===
{retrieved_chunks}
```
